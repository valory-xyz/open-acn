# open-acn (libp2p_node) — Security & Correctness Audit

**Scope:** entire repository (32 Go files + Dockerfile + `run_acn_node_standalone.py`).
**Focus:** security and correctness, both. Whole codebase.
**Method:** parallel deep-dive of each trust boundary (AEA pipe, libp2p streams, delegate TCP, mailbox HTTP, helpers/crypto, monitoring, container), followed by manual verification of all Critical/High findings against source. Findings that did not survive verification have been dropped or downgraded.
**Audit date:** 2026-04-15.

> Severity rubric — **Critical**: remote unauthenticated impact (RCE / message hijack / panic / OOM); **High**: remote DoS, info leak of secrets, race that produces wrong results; **Medium**: local DoS, fragile concurrency, info leak of non-secrets; **Low**: hardening / defence-in-depth; **Info**: notes for follow-up.

## Trust boundaries

The node exposes several distinct attack surfaces. Findings below are grouped by where the untrusted bytes enter:

| Boundary | Authn? | Public by default? | Surface |
|---|---|---|---|
| AEA pipe (TCP `127.0.0.1:port`) | none (loopback) | no | `aea/pipe.go`, `aea/api.go` |
| libp2p stream protocols (`/aea/0.1.0`, `/aea-address/0.1.0`, `/aea-register/0.1.0`) | libp2p secio (peer key) + PoR | yes | `dht/dhtnode/streams.go`, `dht/common/handlers.go`, `dht/dhtpeer/dhtpeer.go` |
| Delegate TCP service | PoR only, **no transport auth/TLS** | yes (binds to `LocalURI` host) | `dht/dhtpeer/dhtpeer.go:handleNewDelegationConnection` |
| Mailbox HTTPS service | bearer Session-Id token | yes | `dht/dhtpeer/mailbox.go` |
| Prometheus monitoring | none | yes (`":port"` ⇒ `0.0.0.0`) | `dht/monitoring/prometheus.go` |
| Container runtime | runs as root | n/a | `Dockerfile` |

## Critical

### C1. Unbounded allocation from a 4-byte length prefix on the AEA pipe
**File:** `aea/pipe.go:54-63`

```go
buf := make([]byte, 4)
_, err := sock.conn.Read(buf)            // 1) short-read not handled
size := binary.BigEndian.Uint32(buf)
buf = make([]byte, size)                 // 2) up to 4 GiB, no cap
_, err = sock.conn.Read(buf)             // 3) short-read not handled
```

Two bugs in one helper:

1. **No size cap.** Anything that can write to the agent loopback port (other local processes, a compromised AEA, a misconfigured port-forward) can request a 4 GiB allocation per frame and OOM-kill the node.
2. **Uses `conn.Read` instead of `io.ReadFull`.** A TCP read may return fewer bytes than requested. Subsequent frames will desynchronise and be parsed as garbage size headers, recursively triggering #1.

The peer-side stream readers in `utils.ReadBytes` *do* have `maxMessageSizeDelegateConnection` (utils/utils.go:750) — the pipe path simply never got the same hardening.

**Fix:** mirror `utils.ReadBytesConn`: use `io.ReadFull`, reject `size > MaxEnvelopeSize` (e.g. 3 MiB), and reject `size == 0`.

### C2. PoR check bypassable when libp2p remote pubkey conversion fails
**File:** `dht/common/handlers.go:78-84`

```go
remotePubkey, err := utils.FetchAIPublicKeyFromPubKey(stream.Conn().RemotePublicKey())
ignore(err)                             // <- silently swallowed
status, err := dhtnode.IsValidProofOfRepresentation(
    aeaEnvelope.Record,
    aeaEnvelope.Record.Address,
    remotePubkey,                       // empty string on err
)
```

`IsValidProofOfRepresentation` (`dht/dhtnode/utils.go:75`) compares `record.PeerPublicKey != representativePeerPubKey`. If the conversion errors, `remotePubkey` is `""` and an attacker who sends a record with `PeerPublicKey = ""` passes the equality check. Signature verification on line 92 then runs `VerifyLedgerSignature(ledger, []byte(""), record.Signature, record.PublicKey)` — it verifies that `record.PublicKey` signed the empty string, which an attacker can produce trivially with their own key.

End result: any record whose `PeerPublicKey` is empty and whose `Signature` is a valid signature by `PublicKey` over `""` is accepted as a valid PoR for *any agent address* the attacker chooses. Combined with the lookup path (`dht/dhtclient/dhtclient.go:572`, same helper), this enables **address impersonation and message hijack**.

Realistic trigger condition: `FetchAIPublicKeyFromPubKey` is the only failure mode. It fails on unsupported key types — currently `RSA`, `Ed25519`, etc. (not the secp256k1 used by `noise`/`secio` peers). The trigger may be hard against current libp2p config, but it is *defence-in-depth absent* and would silently break the moment another transport key type is enabled.

**Fix:** treat the conversion error as fatal: `if err != nil { stream.Reset(); return }`. Also reject empty `record.PeerPublicKey` and empty `record.Signature` in `IsValidProofOfRepresentation` regardless.

### C3. Out-of-bounds panic on a short Ethereum signature
**File:** `utils/utils.go:466` (also `utils/utils.go:278` and `utils/utils.go:569` — `publicKey[2:]` without `len >= 2` check)

`RecoverAddressFromEthereumSignature` indexes `sigBytes[64]` with no length guard. Any caller that feeds attacker-controlled bytes into Ethereum signature recovery — anywhere in the PoR/registration path that might use `ledgerId == "ethereum"` — gives the attacker a one-packet remote panic.

**Fix:** `if len(sigBytes) < 65 { return "", errors.New("signature too short") }` at the top of the function. Audit every `s[N:]` / `s[N]` slice in `utils/utils.go` for the same pattern.

---

### C4. Mailbox session token is a bearer token with no binding to the TLS client
**File:** `dht/dhtpeer/mailbox.go:205-219` (and registration around `mailbox.go:55-80`)

`apiGetEnvelope` resolves the `Session-Id` header against `mailboxServer.sessions`, returns the mapped agent's mailbox, and pops the head envelope. The token is:

- generated server-side at `/register`,
- transmitted over TLS,
- but **not bound** to the registering client's identity (TLS client cert, peer pubkey, IP, or PoR signature).

Anyone who obtains a Session-Id (proxy logs, browser history, server-side log capture, MITM of a client that disabled cert verification, leaked from a debug dump, …) can read **and drain** the entire mailbox of that agent. The TODO at `mailbox.go:55` ("double register fix!!!") confirms this area is known to be fragile: the second `/register` for an address overwrites the first record without revoking the prior session.

**Fix:** bind the session to the registering peer (e.g. require the client to sign each request with the agent key declared at registration; or require client TLS cert and pin to the cert presented at register-time). At minimum, invalidate prior sessions on re-register and rate-limit `Session-Id` enumeration.

## High

### H1. Delegate TCP service has no read/write deadlines (slowloris)
**File:** `dht/dhtpeer/dhtpeer.go:853-1045` (`handleNewDelegationConnection`)

Every accepted delegate connection spawns a goroutine that calls `acn.ReadAgentRegistrationMessage` → `utils.ReadBytesConn` with no `SetReadDeadline`. An attacker can `Accept`-and-stall N connections, each pinning at least one goroutine and one `make([]byte, size)` buffer for the size header. Connection count is also unbounded (no `MaxConnections` semaphore).

**Exploit:** trivial slowloris from anywhere the delegate port is reachable (and the default bind is whatever `LocalURI` host is — easy to land on `0.0.0.0` in a container).

**Fix:** `conn.SetDeadline(time.Now().Add(N))` on accept; per-IP and global connection limits; idle-timeout the per-connection goroutine.

---

### H2. Per-(To,From) routing goroutines and `syncMessages` map leak across the connection lifetime
**File:** `dht/dhtpeer/dhtpeer.go:996-1045`

For each unique `(To, From)` pair seen on a delegate connection, a channel is added to `syncMessages` and a routing goroutine is spawned. On connection close (`dhtpeer.go:1037-1044`) `tcpAddresses` and `acnStatuses` are cleaned up but `syncMessages` is not — neither the channel nor the goroutine is reaped. A long-lived attacker (or a buggy client) that cycles through many `From` values leaks one goroutine + one channel per pair, monotonically.

**Fix:** track which `syncMessages` keys belong to which connection and delete + close on disconnect; ensure the routing goroutine exits when its input channel is closed.

---

### H3. `acnStatuses` send is blocking — slow client stalls the routing goroutine
**File:** `dht/dhtpeer/dhtpeer.go:1900-1906` (`AddAcnStatusMessage`)

`acnStatuses[addr]` is a fixed-size buffered channel. The send is unconditional. A delegate client that stops reading status updates fills the buffer; subsequent sends block the goroutine that called `RouteEnvelope`, which holds upstream resources. One slow client → cascading head-of-line blocking.

**Fix:** non-blocking `select { case ch <- msg: default: drop+log }`, or per-channel timeout.

---

### H4. Slow mailbox client holds the *global* mailbox lock across network I/O
**File:** `dht/dhtpeer/mailbox.go:207-237`

`apiGetEnvelope` does `mailboxServer.lock.Lock(); defer mailboxServer.lock.Unlock()` and then `res.Write(buf)` while holding the lock. `RouteEnvelope` (`mailbox.go:299-323`) and every other handler take the same lock. A single slow HTTP reader stalls registration, send, and receive **for every agent** on this mailbox.

(Note: the earlier sub-agent claim of TOCTOU on this handler was wrong — the lock *is* held across the read+delete. The real bug is that it's also held across network I/O.)

**Fix:** marshal the envelope under the lock, drop the lock, then write to the network. Re-acquire only to delete the head if the write succeeded (and re-validate that the head is the same envelope).

---

### H5. Race on `closing` flag and double-close of channels in the AEA API
**File:** `aea/api.go:85,164-168,349,375,398,420`

`closing` is a plain `bool` mutated by `Stop()` and read by `listenForEnvelopes` and `envelopeSendLoop` with no synchronisation. Channels (`out_queue`, `send_queue`) are closed unconditionally in `Stop()` with no `sync.Once`. Concurrent shutdown paths (signal handler + pipe-error path → `Stop()` from the listen goroutine) produce both a data race and a `close of closed channel` panic.

**Fix:** convert `closing` to `atomic.Bool`; wrap the close block in `sync.Once`.

---

### H6. Private key material logged at debug level
**File:** `aea/api.go:218` — `logger.Debug().Msgf("id: %s", aea.id)` where `aea.id` is the AEA private key from `AEA_P2P_ID`.
**Also:** `acn/utils.go:187` — logs raw envelope bytes that may contain credentials.

Debug logs are routinely captured by aggregation pipelines, container log drivers, and crash dumps. A node that ships logs off-box silently exfiltrates the operator key the moment debug logging is enabled.

**Fix:** never log secret material. If a debug fingerprint is needed, log a hash prefix (`fmt.Sprintf("%x", sha256.Sum256(key))[:8]`).

---

### H7. ACN error responses echo internal parser error strings to the network
**File:** `acn/utils.go:215, 278` (already flagged with `// TOFIX(LR) ... potentially a security vulnerability`); reflected at `aea/utils.go:216,279,441` and `dht/common/handlers.go:115,156`.

`SendAcnError(pipe, err.Error(), …)` returns Go-style error strings ("proto: cannot parse … wire type X at offset Y") to the remote peer, simplifying protocol fingerprinting and fuzzing. The fact that the comment says "TOFIX … potentially a security vulnerability" and the code still does it is the tell.

**Fix:** send a fixed code + a short generic message; log the detailed error locally.

---

### H8. Delegate registration: address ownership trust depends entirely on PoR — no rate limit, no replay nonce
**File:** `dht/dhtpeer/dhtpeer.go:1757-1850`

Once an attacker has a copy of any agent's PoR bundle (e.g. obtained from a public lookup over the DHT, leaked from a peer's persistent storage, recovered from a packet capture), they can present it to *any* delegate or relay endpoint and become the route for that address until the legitimate agent re-registers. There's no nonce, no challenge-response, no expiry.

This is a protocol-level issue, not a code bug, but every code path that takes an `AgentRecord` from the network treats it as a stable bearer credential. Combined with C2 it is severely exploitable; even without C2 it makes PoRs sensitive material that must never be persisted in the clear or returned via `/lookup`.

**Fix:** the registration message should include a fresh server-issued nonce signed by the agent key; PoRs should not be returned to arbitrary lookups by content.

---

### H9. Persistent agent-record storage grows without bound and is read entirely into memory at startup
**File:** `dht/dhtpeer/dhtpeer.go:459-537`

`saveAgentRecordToPersistentStorage` appends; `initAgentRecordPersistentStorage` reads all records into a map at startup. With churn, the file grows unboundedly; on restart it must all fit in RAM. No rotation, no compaction, no max-records cap.

**Fix:** size-cap + log rotation (e.g. SQLite or LMDB) or keyed file with periodic compaction.

---

### H10. Prometheus endpoint binds to all interfaces with no auth
**File:** `dht/monitoring/prometheus.go:122` — `":" + port`.

Metrics encode network topology (peer counts, peer addresses if labelled), latency distributions, and traffic volumes — useful reconnaissance. There is also no label-cardinality safeguard: a future change that adds an `agent_address` label silently turns the histogram into an unbounded-cardinality memory sink.

**Fix:** bind to `127.0.0.1` by default; require explicit opt-in for public exposure; document not to use per-agent labels.

---

### H11. Container runs as root, downloads Go toolchain over network without checksum
**File:** `Dockerfile:3, 5, 10`

- `USER root` and entrypoint runs as root → any RCE in the node is also root in the container, with full filesystem and network access.
- `wget https://dl.google.com/go/go1.24.0.linux-amd64.tar.gz` with no checksum/signature verification → a compromised mirror or MITM injects a backdoored Go compiler into every build.
- `apt-get upgrade -y` and `pip install pymultihash ecdsa base58` with no version pinning → reproducibility and supply-chain risk.

**Fix:** `USER acn` (uid:gid != 0) before the entrypoint; `RUN echo "<sha256> go1.24.0.linux-amd64.tar.gz" | sha256sum -c -`; pin pip and apt versions; consider `pip install --require-hashes`.

## Medium

### M1. `monitoring` map dictionaries are accessed without synchronisation
**Files:** `dht/monitoring/prometheus.go:34-95` (`counterDict`/`gaugeDict`/`histoDict`); `dht/monitoring/file.go:214-226` (iteration over `gaugeDict`/`counterDict` while concurrent writers may exist).

Concurrent map access in Go is undefined behaviour — the runtime will detect it and crash the process (`fatal error: concurrent map writes`).

**Fix:** add `sync.RWMutex` (the `Timer` type already does this).

---

### M2. Monitoring stats file written world-writable
**File:** `dht/monitoring/file.go:192` — `os.OpenFile(fm.path, os.O_WRONLY|os.O_CREATE, 0666)`. Other local users can corrupt the file or read operational data.

**Fix:** `0600`.

---

### M3. `os.Getwd` error swallowed when constructing the stats file path
**File:** `dht/monitoring/file.go:135-136` — `cwd, _ := os.Getwd()`. If it fails, files are written under `/`.

**Fix:** check the error.

---

### M4. `slow_queue` close races with sender on shutdown
**File:** `dht/dhtpeer/dhtpeer.go:653, 693, 1207, 1230-1250`

`isClosing` is read in the routing path and written in `Close()` with no atomic; the slow queue is closed unconditionally. A concurrent send produces a `send on closed channel` panic during shutdown.

**Fix:** atomic flag + drain + `sync.Once` around close.

---

### M5. Reuse of a single bootstrap context across the retry loop
**File:** `dht/dhtclient/dhtclient.go:374-399` — context with `bootstrapTimeout` is captured once and passed into each retry; later attempts inherit a near-expired deadline and spuriously fail.

**Fix:** create a fresh per-attempt context derived from the parent deadline.

---

### M6. Errors silently ignored on critical parse paths
**Files:** `utils/utils.go:252` (`hex.DecodeString` for public key), `utils/utils.go:508` (`btcec.PrivKeyFromBytes`), `aea/api.go:246, 269, 285, 301` (port `ParseUint`), `aea/pipe.go:86` (port `ParseUint`).

In each case a malformed input becomes a silently-zero value or nil pointer that surfaces as a confusing failure later (or, for the crypto helpers, as an apparently-valid but unusable key passed to libp2p).

**Fix:** propagate the error.

---

### M7. `ReadBytes`/`WriteBytes` panic on nil stream
**File:** `utils/utils.go:734-736, 774-776, 800-802`

Defensive panic at a layer this deep should be a returned error — a logic bug at the call site becomes a process crash.

**Fix:** return an error.

---

### M8. `FileHistogram.Observe` is bucketing incorrectly
**File:** `dht/monitoring/file.go:95-106`

The loop counts every bucket whose threshold is `<= value` instead of placing the observation in a single bucket. Histograms reported via the file backend will be inconsistent with Prometheus semantics.

**Fix:** standard cumulative-bucket logic (`for i; v > buckets[i]; i++` then increment exactly one bucket).

---

### M9. Open relay with no per-peer bandwidth quota
**File:** `dht/dhtpeer/dhtpeer.go` (relay enable path around `EnableRelayService`).

Any peer can register as a relay client and send unlimited traffic. Useful for legitimate clients behind NATs, but identical to an open mail relay if exposed to the public internet.

**Fix:** per-peer rate limit / token bucket; configurable allow-list.

---

### M10. `run_acn_node_standalone.py`: line-based config writer accepts unescaped `=` and newlines
**File:** `run_acn_node_standalone.py:105-107`

`AcnNodeConfig.dump()` writes `key=value` lines without escaping. A value (often sourced from environment) containing `\n` or `=` corrupts the file and can be parsed back as an unrelated key.

**Fix:** validate/escape, or switch to JSON/YAML with strict loaders.

## Info / not-bugs (downgraded after verification)

- **Mailbox `apiGetEnvelope` TOCTOU.** Initial finding from one sub-agent claimed read-then-delete was racy. The handler does in fact hold `mailboxServer.lock` for the entire body via `defer Unlock`. The real issue (lock held across network I/O) is **H4** above.
- **`IsValidProofOfRepresentation` doesn't bind PeerPublicKey.** The validator at `dht/dhtnode/utils.go:75` *does* check `record.PeerPublicKey == representativePeerPubKey`. The exploitable bug is the *caller* in `dht/common/handlers.go:79` ignoring the error from `FetchAIPublicKeyFromPubKey` (covered as **C2**), not the validator itself.
- **Hardcoded `127.0.0.1` in `pipe.go`.** This is a deliberate loopback boundary, not a vulnerability.

## Suggested remediation order

1. **C1** (cap pipe size + `io.ReadFull`) — tiny patch, removes a one-packet OOM.
2. **C3** (`len(sig) >= 65`) — one-line patch, removes a one-packet panic.
3. **C2** (handle the `FetchAIPublicKeyFromPubKey` error in `handlers.go`) — closes the PoR-bypass path even if currently hard to trigger; also reject empty-string fields in `IsValidProofOfRepresentation`.
4. **H1, H2, H3** (delegate deadlines, goroutine/map cleanup on disconnect, non-blocking status sends) — together they remove the easiest remote DoS.
5. **C4 + H8** (mailbox bearer-token binding, registration nonce) — protocol-level work; design before implementing.
6. **H5, H6, H7, H10, H11, M1–M4** — straightforward cleanup, mostly local diffs.
7. The remaining Mediums and Lows as time permits.
